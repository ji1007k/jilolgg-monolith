# 🚀 Future Work: 시스템 확장 및 고도화 계획

현재의 **FE/BE 통합 모놀리스 구조**에서 데이터 처리량이 급증할 경우를 대비하여, 코드의 전면 수정 없이 시스템을 수평 확장(Scale-out)하기 위한 단계적 고도화 방안입니다.

## 1. Redis Streams 기반 분산 수집 파이프라인 (Distributed Worker)
단일 인스턴스에 집중된 외부 API 수집 부하를 여러 노드로 분산하여 처리 효율을 극대화할 계획입니다.

- **Master-Worker 구조**: 마스터 노드가 수집 대상 리그 ID를 **Redis Streams**에 발행(Publish)하면, 여러 워커 인스턴스가 이를 수신하여 병렬로 처리합니다.
- **중복 작업 방지**: Redis의 **Consumer Group** 기능을 활용하여, 동일 인스턴스 구조 내에서도 특정 데이터 수집 작업이 단 한 번만 수행되도록 보장합니다.
- **장애 복구(Reliability)**: 작업 중인 인스턴스가 중단될 경우, Redis의 **PEL(Pending Entries List)**과 **XAUTOCLAIM**을 활용하여 다른 인스턴스가 해당 작업을 자동으로 회수 및 재처리하도록 구현할 예정입니다.

## 2. Spring Profile을 활용한 인스턴스 역할 분리
인스턴스 확장 시 프론트엔드(Next.js)와 백엔드가 동시에 복제되는 리소스 낭비를 제어하기 위한 전략입니다.

- **역할별 프로필 운영**: 동일한 배포 아티팩트를 사용하되, 실행 시점에 `spring.profiles.active` 설정에 따라 특정 기능을 활성화/비활성화합니다.
- **Web Node**: 사용자 응답(UI/API) 및 실시간 채팅 서비스 전담.
- **Worker Node**: Redis Streams 컨슈머와 배치 스케줄러만 활성화하여 데이터 수집 및 정제 전담.
- **기대 효과**: 배포 프로세스의 복잡도는 유지하면서, 배치 전용 노드의 메모리 및 CPU 자원을 비즈니스 로직 처리에 집중시킬 수 있습니다.

## 3. 리소스 기반 동적 스레드풀 관리
작업 부하에 따라 시스템 자원을 유연하게 할당하는 메커니즘을 도입할 예정입니다.

- **동적 풀 사이즈 조절**: 현재 시스템의 Heap 메모리 상태와 CPU 사용률을 모니터링하여, 동기화 작업에 할당되는 **Core Pool Size**를 실시간으로 조정함으로써 서비스 가용성을 선제적으로 확보합니다.
