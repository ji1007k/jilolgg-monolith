# 부하 테스트 보고서
## 테스트 환경
- **서버**: AWS EC2 프리티어 t2.micro (1GB RAM)
- **애플리케이션**: Spring Boot + Redis Cache
- **테스트 도구**: Apache JMeter
- **API**: GET /api/lol/matches (126KB 응답 크기)
- **캐싱**: Spring Cache + Redis 적용됨

## 주요 관찰 사항

### 1. 캐시 효과 확인
- **Redis 모니터링**: `redis-cli monitor`로 캐시 히트 확인
- 동일한 키로 반복 GET 요청 시 DB 조회 없이 Redis에서 응답
- 분산 락(Redisson) 정상 동작 확인

### 2. 성능 특이점
- **150명 (Loop 5)**: Throughput 2.4/sec로 급락 → Loop Count의 중요성 확인
- **250명**: 응답시간이 오히려 개선 (57ms) → JVM 최적화 또는 캐시 워밍업 효과
- **350명**: Throughput이 29.2/sec로 최고치 기록

### 3. 서버 안정성
- **모든 테스트에서 Error % 0-0.5%**: 매우 안정적
- **응답시간**: 39-67ms 범위로 일관성 유지
- **95% Line**: 200ms 이하로 대부분 요청이 빠르게 처리

## 성능 분석

### Throughput 변화 패턴
```
20명: 10.3/sec → 100명: 16.7/sec → 200명: 25.0/sec → 350명: 29.2/sec
```
- 200-250명 구간에서 잠시 정체
- 350명에서 최고 성능 달성

### 응답시간 안정성
- **평균 응답시간**: 39-67ms (매우 우수)
- **95% 응답시간**: 93-229ms (양호)
- GET 요청 + 캐시 조합으로 일관된 성능 유지

## 결론 및 다음 단계

### 현재 평가
- **t2.micro 1GB RAM**으로 **350명 동시 접속** 처리 가능
- **초당 29.2개 요청** 처리 (일일 252만 요청 가능)
- Error Rate [**0%**]로 안정성 확인

### 한계점 미도달 이유
1. **GET 요청의 가벼움**: 읽기 전용, 캐시 효과
2. **작은 응답 크기**: 126KB
3. **Redis 캐싱**: DB 조회 없이 메모리에서 응답

### 추가 테스트 권장사항
1. **POST 요청 테스트**: 실제 쓰기 작업 부하
2. **혼합 시나리오**: GET 80% + POST 15% + PUT/DELETE 5%
3. **캐시 없는 API**: 복잡한 DB 쿼리 테스트
4. **더 큰 부하**: 500-1000명 동시 접속

### 비용 고려사항
- **AWS 프리티어**: 월 15GB 아웃바운드 데이터 전송 제한
- **현재 사용량**: 시간당 약 7.6GB (126KB × 16.7/sec × 3600)
- **권장**: 짧은 시간 테스트로 제한, 장시간 테스트 시 과금 주의
