# 2026-02-24 프로젝트 수정 및 버그 수정 내역

## 1. 프론트엔드(FE) - 백엔드(BE) 통합 작업 내역

### 프론트엔드 빌드 및 이식
*   **구조 통합**: Next.js 프로젝트를 `be/frontend` 폴더로 이동.
*   **빌드 자동화**: Gradle의 `node-gradle` 플러그인을 활용하여 `./gradlew build` 시 프론트엔드가 자동으로 빌드되고, 결과물(`out`)이 Spring Boot의 정적 리소스 경로(`src/main/resources/static/jikimi/`)로 복사되도록 설정.
*   **Next.js 설정**: `next.config.mjs`에 `output: 'export'` 및 `basePath: '/jikimi'`를 적용하여 정적 배포 최적화.

### 라우팅 및 리소스 서빙
*   **HomeController**: 루트(`/`, `/home`) 및 `/jikimi` 접속 시 `/jikimi/`로 자동 리다이렉트되도록 설정.
*   **WebConfig**: `/jikimi/**` 경로로 들어오는 요청을 정적 파일로 서빙하며, SPA(Single Page Application) 라우팅을 위해 파일이 없을 경우 `index.html`을 반환하도록 설정. (단, 확장자가 있는 파일 요청 실패 시에는 Hydration 에러 방지를 위해 index.html을 주지 않도록 개선)
*   **ApiPrefixFilter**: 프론트엔드에서 일괄적으로 사용하는 `/api` 접두사를 서버 내부에서 투명하게 제거하기 위해 `HttpServletRequestWrapper` 기반의 필터 구현. 이를 통해 보안 설정이나 컨트롤러 코드를 수정하지 않고도 `/api/lol/leagues` 요청을 `/lol/leagues`로 매핑.

### 보안 및 인증 (Spring Security)
*   **SecurityConfig**: `/jikimi/**` 및 공개 API들에 대해 `permitAll()` 설정.
*   **CustomJwtFilter**: 쿠키 기반 JWT 인증 로직 구현. 토큰이 없거나 유효하지 않아도 공개 경로 접근을 방해하지 않도록 비파괴적(Non-destructive) 방식으로 개선.
*   **Local Dev 환경 최적화**: `application-dev.properties`에서 `cookie.secure=false`, `cookie.same-site=Lax`로 설정하여 HTTP 환경에서도 크롬 등 브라우저가 JWT 쿠키를 정상적으로 전송하도록 수정.

| :--- | :--- | :--- | :--- |
| **1** | **로그인 후 정적 리소스 401/404** | `/jikimi/index.html`로 직접 리다이렉트 시 상대 경로 참조 오류 발생. -> `/jikimi/` (슬래시 포함)로 리다이렉트하도록 수정하여 Next.js basePath 인식 정상화. | **조치 완료** |
| **2** | **React Hydration 에러 (#418)** | 서버에 없는 정적 파일(예: `.json`, `.txt`) 요청 시 `index.html`이 대신 반환되어 리액트가 해석 오류를 일으킴. -> `WebConfig`에서 확장자가 있는 요청은 폴백하지 않도록 수정. | **조치 완료** |
| **3** | **`jikimi.txt` 401 에러** | Next.js App Router가 사용하는 특수 파일이 보안 설정에 의해 차단됨. -> `SecurityConfig`에 `/jikimi.txt` 직접 허용 추가. | **조치 완료** |
| **4** | **리그 정렬 순서 변경 불가** | `/api` 제거 후 포워딩 과정에서 CSRF 토큰 검증이나 인증 정보가 누락되는 문제. -> `ApiPrefixFilter`를 `chain.doFilter` 방식으로 변경하여 Security 필터들이 인지할 수 있게 수정. | **조치 완료** |
| **5** | **로그인 후 웹소켓 연결 실패** | `JwtHandshakeInterceptor`가 `Jwt` 객체만 처리하고 `CustomUserDetails`를 처리하지 못해 `ClassCastException` 발생. -> 다양한 인증 주체(Principal)를 지원하도록 핸들러 로직 보완. | **조치 완료** |
| **6** | **메인 로고 클릭 시 에러** | 로고 링크가 `/jikimi/index.html` 등 확장자가 포함된 경로를 가리켜 리소스 매핑 충돌 발생. -> `HomeController`에서 모든 루트 진입점을 `/jikimi/`로 통일. | **조치 완료** |

---
**주의 사항**: 스프링 시큐리티 설정(`SecurityConfig`)에서 특정 URL을 무분별하게 `permitAll()` 하는 대신, `ApiPrefixFilter`를 통해 원래의 경로 설정을 그대로 활용할 수 있도록 구조적으로 해결함.

## 3. FE 빌드 및 BE 이식 가이드 (Gradle 연동)

프론트엔드 수정 사항을 백엔드에 반영하고 실행하는 방법입니다. 이제 프로젝트가 `jilolgg` 루트로 통합되었습니다.

### 빌드 및 배포 순서
모든 작업은 프로젝트 루트 디렉토리(`jilolgg/`)에서 수행합니다.

1.  **전체 빌드 (FE 빌드 + 리소스 복사 + BE 컴파일)**
    ```bash
    ./gradlew build
    ./gradlew build -x test --info
    ```
    *   이 명령어를 실행하면 `frontend/`에서 `npm run build`가 실행되고, 결과물(`out`)이 자동으로 `src/main/resources/static/jikimi/` 경로로 복사됩니다.

2.  **프론트엔드 작업만 반영할 경우**
    ```bash
    ./gradlew copyFrontend
    ```
    *   `buildFrontend` (FE 빌드)와 `copyFrontend` (파일 복사) 작업만 빠르게 수행합니다.

3.  **애플리케이션 실행/종료**
    ```bash
    ./gradlew bootRun
    ./gradlew --stop
    ```

### 주요 경로 정보
*   **FE 소스 코드**: `frontend/`
*   **FE 빌드 결과물 (임시)**: `frontend/out/`
*   **BE 정적 리소스 최종 위치**: `src/main/resources/static/jikimi/`
    *   Spring Boot는 빌드 시 이 경로의 파일들을 읽어 `/jikimi/**` 경로로 서비스합니다.

### 연동 원리 (`build.gradle`)
*   `com.github.node-gradle.node` 플러그인을 사용하여 Gradle 내에서 Node.js 환경을 제어합니다.
*   `buildFrontend` 테스크: `npm install` 및 `npm run build`를 수행합니다.
*   `copyFrontend` 테스크: `frontend/out`의 내용을 `static/jikimi`로 복사하며, 기존 파일을 정리(clean)하여 찌꺼기가 남지 않게 합니다.
