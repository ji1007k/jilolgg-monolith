# 면접 대비 트레이드오프 정리

## 1. Spring Batch Partitioning

### 선택
- 단일 스레드 → Partitioning 병렬 처리

### 트레이드오프
- 👍 처리량 증가
- 👎 **DB 커넥션 / 외부 API 동시 호출 증가**

### 실제 리스크
- 커넥션 풀 고갈
- API rate limit
- 락 경합 증가

### 한 줄 정리
👉  
“처리 속도는 개선됐지만, 병렬 처리로 인해 DB 및 외부 API 부하를 고려해 파티션 수를 제한”

---

## 2. Redis 분산 락 (Redisson)

### 선택
- DB 비관적 락 → Redis 분산 락

### 트레이드오프
- 👍 DB 부하 감소 / 확장성
- 👎 **네트워크 의존 + 락 유실 가능성**

### 실제 리스크
- Redis 장애 시 락 깨짐
- TTL 만료 타이밍 문제
- 완벽한 strong consistency 아님

### 한 줄 정리
👉  
“DB 락 대비 확장성은 확보했지만, Redis 장애 및 TTL 기반 락 유실 가능성을 고려”

---

## 3. Cache (Look-aside + Evict)

### 선택
- DB 직접 조회 → 캐시 사용

### 트레이드오프
- 👍 조회 성능 개선
- 👎 **Stale Data 가능성**

### 실제 리스크
- Evict 타이밍 전에 조회
- 동시성 상황에서 캐시 불일치

### 한 줄 정리
👉  
“성능 향상을 위해 캐시를 사용했지만, 데이터 정합성을 위해 Evict 전략과 TTL을 함께 고려”

---

## 4. Dedupe (조회 레이어 병합)

### 선택
- DB 통합 → 조회 시 병합

### 트레이드오프
- 👍 원본 데이터 보존 / 유연성
- 👎 **조회 로직 복잡도 증가**

### 실제 리스크
- 쿼리 복잡도 증가
- 성능 저하 가능
- 버그 발생 포인트 증가

### 한 줄 정리
👉  
“데이터 정합성을 유지하기 위해 조회 시 병합을 선택했지만, 조회 로직 복잡도가 증가”

---

## 5. 모놀리스 배포 (Next.js + Spring Boot)

### 선택
- FE/BE 분리 → 단일 배포

### 트레이드오프
- 👍 운영 단순화 / CORS 제거
- 👎 **독립 배포 불가 / 확장성 제한**

### 실제 리스크
- 프론트만 수정해도 전체 배포
- 트래픽 증가 시 분리 필요

### 한 줄 정리
👉  
“운영 단순성을 위해 모놀리스 구조를 선택했지만, 향후 트래픽 증가 시 서비스 분리를 고려”

---

## 6. Redis 캐시 + 락 같이 사용

### 선택
- Redis 하나로 캐시 + 락 처리

### 트레이드오프
- 👍 인프라 단순화
- 👎 **Redis 의존도 집중 (Single Point of Failure)**

### 실제 리스크
- Redis 죽으면:
  - 캐시도 날아감
  - 락도 깨짐

### 한 줄 정리
👉  
“운영 단순화를 위해 Redis를 통합 사용했지만, 장애 시 영향 범위가 커지는 점을 고려”

---

## 7. Spring Batch vs 실시간 처리

### 선택
- Batch 기반 동기화

### 트레이드오프
- 👍 안정적 / 재처리 용이
- 👎 **실시간성 부족**

### 실제 리스크
- 데이터 지연 발생
- 사용자 최신 데이터 불일치

### 한 줄 정리
👉  
“안정적인 처리를 위해 배치를 선택했지만, 실시간성이 필요한 경우 이벤트 기반 구조를 고려”

---

## 핵심 요약 (면접용)

👉  
“전체적으로 처리 성능과 운영 단순성을 우선했기 때문에 정합성 지연, Redis 의존성, 확장성 제한 같은 트레이드오프가 존재합니다. 각 요소는 TTL, 파티션 수 조정, 향후 서비스 분리 등을 통해 보완 가능한 구조로 설계했습니다.”
